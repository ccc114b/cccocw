# 虛擬機技術介紹：從程式人的視角

## 什麼是虛擬機？

虛擬機（Virtual Machine, VM）是一種軟體技術，它能在同一台硬體上模擬出多個獨立的作業系統環境。對程式人而言，虛擬機就像是有魔法的工作站——你可以在一台筆電上同時執行 Linux、Windows、macOS，而且它們彼此隔離，互不干擾。

### 核心概念：Hypervisor

Hypervisor（虛擬機器監視器）是虛擬機的核心元件，負責管理硬體資源並在多個 Guest OS 之間分配 CPU、記憶體、磁碟和網路。

**Type 1 Hypervisor（原生虛擬化 / Bare Metal）**：直接運行在硬體上，不依賴任何主機作業系統，效能最高，常用於伺服器環境。

- **VMware ESXi**：商業級伺服器虛擬化的標竿，vSphere 套件的核心，支援 vMotion、HA、DRS 等企業級功能
- **Microsoft Hyper-V**：Windows Server 內建的虛擬化平台，也支援 Linux Guest OS，與 Windows 生態整合良好
- **Xen**：開源的超管理器，Amazon AWS 早期採用的技術，安全性高，支援 Paravirtualization
- **KVM**（Kernel-based Virtual Machine）：Linux 核心內建的虛擬化模組，本質上是 Linux 的一部分，現在是最流行的開源虛擬化方案
- **Proxmox VE**：基於 Debian 的開源虛擬化平台，整合了 KVM 和 LXC，提供 Web UI 管理

**Type 2 Hypervisor（寄宿虛擬化）**：運行在主機作業系統上的應用程式，例如 VirtualBox、VMware Workstation。

## VirtualBox、QEMU、Docker：技術差異解析

這三個工具雖然都常被拿來做「虛擬化」，但它們的定位和技術實現有根本性的差異。

### VirtualBox

**定位**：Type 2 Hypervisor，桌面級虛擬化解決方案。

VirtualBox 是 Oracle 開發的開源虛擬化軟體，專為開發者和測試環境設計。它的設計目標是「讓開發者能在本機輕鬆建立 VM」，所以強調易用性和 UX。

**技術特點**：
- 基於 QEMU 的變體改造而來
- 提供 Guest Additions 增強工具，提升 Guest OS 與 Host 的整合（共享資料夾、剪貼簿共享、解析度自動調整）
- 支援「無縫模式」，讓 VM 內的視窗與 Host 桌面融為一體
- 硬體模擬：vCPU、vRAM、v磁碟、v網卡、v音效卡

**適合場景**：
- 開發者需要測試跨平台軟體
- 需要同時運行多個不同 OS 的測試環境
- 學習作業系統或進行實驗

**程式人觀點**：VirtualBox 像是「入門級重機」——好上手、功能齊全，但效能不是頂尖。對於日常開發測試已經足夠。

### QEMU

**定位**：硬體模擬器 + 虛擬化加速器（配合 KVM）。

QEMU 是純軟體的 CPU 和硬體模擬器，它的厲害之處在於：它能模擬 x86、ARM、MIPS、PowerPC 等各種架構。這意味著你可以在 x86 Linux 上模擬一台 ARM 的 Android 設備。

**技術特點**：
- **模擬模式**：純軟體翻譯，Guest 的每條指令都會被 QEMU 翻譯成 Host 的指令，效能較慢
- **KVM 加速**：搭配 Linux 內建的 KVM（Kernel-based Virtual Machine），可以直接利用硬體虛擬化指令（Intel VT-x、AMD-V），效能大幅提升
- 支援熱遷移（Live Migration）、snapshot、儲存後端多樣化
- 沒有 GUI，需要用 command line 或 libvirt 介面（如 virsh、virt-manager）

**適合場景**：
- 嵌入式開發（模擬 ARM/MIPS 開發板）
- 跨平台測試
- 需要高彈性的伺服器虛擬化
- 學習作業系統核心（QEMU 是 Linux Kernel 開發者的標準工具）

**程式人觀點**：QEMU 是「瑞士刀」——功能強大但需要時間熟悉。Linux Kernel 開發者一定會遇到它，因為 QEMU 是 upstream 的標準模擬器。

### Docker

**定位**：作業系統層級的容器化技術，不是傳統意義的虛擬機。

Docker 這個名字現在幾乎和「容器」畫上等號，但它的核心技術是 **Container**，不是 VM。這個差異是理解 Docker 的關鍵。

**技術特點**：
- **共享 Host Kernel**：所有 containers 共享主機的 Linux Kernel，不需要模擬整個作業系統。這是 Docker 與 VirtualBox/QEMU 最根本的差異
- **Namespaces & Cgroups**：Linux 核心功能，用於隔離行程、網路、檔案系統、資源限制
- **Overlay Filesystem**：使用 Union FS（OverlayFS、aufs）實現多層映象，節省磁碟空間
- **Image 機制**：類似 Git 的分層概念，可以從基礎映象層層疊加，映象可共享、復用
- ** lightweight**：啟動速度以秒計算，記憶體佔用遠低於 VM

**適合場景**：
- 應用程式打包與部署（DevOps）
- 微服務架構
- 持續整合/持續部署（CI/CD）
- 多版本開發環境（如同時運行 Node.js 12、14、18）

**程式人觀點**：Docker 的本質是「環境一致性」問題的解決方案。開發者常說「在我的機器上能跑啊」，Docker 就是為了消滅這句話而生的。

## 從程式碼看虛擬化原理

### 1. QEMU 的指令模擬：Tiny Code Generator (TCG)

QEMU 的核心是 TCG（Tiny Code Generator），它會把 Guest 指令翻譯成 Host 指令。偽コード如下：

```c
// QEMU 模擬 CPU 執行的概念
while (1) {
    // 1. 讀取 Guest 的下一條指令
    target_ulong guest_pc = cpu->pc;
    uint32_t instr = read_guest_memory(cpu, guest_pc);
    
    // 2. 解譯 Guest 指令（翻譯成中間表示 IR）
    TCGOp *ir = decode_instruction(instr);
    
    // 3. 將 IR 編譯成 Host 指令
    TCGOp *host_code = tcg_compile(ir);
    
    // 4. 執行翻譯後的 Host 指令
    execute_host_code(host_code);
    
    // 5. 更新 Guest 狀態（PC、暫存器等）
    cpu->pc += 4;
}
```

這就是為什麼 QEMU 在純模擬模式下慢的原因——每條 Guest 指令都要經過「解釋→編譯→執行」的過程。

### 2. KVM 的硬體輔助虛擬化

當 QEMU 搭配 KVM 使用時，流程變成：

```c
// KVM 模式下，QEMU 將大部分指令直接交給硬體執行
int kvm_run(KVMState *s, CPUState *cpu) {
    // ioctl 進入 VMX non-root mode
    ioctl(s->fd, KVM_RUN, 0);
    
    // 硬體會自動處理大部分指令
    // 只有特殊指令（hypercalls、I/O）會退出（VM Exit）
    switch (cpu->exit_reason) {
        case KVM_EXIT_IO:
            // Guest 執行 IN/OUT 指令，模擬 I/O
            handle_io(cpu);
            break;
        case KVM_EXIT_HLT:
            // Guest 執行 HLT，CPU 進入閒置
            handle_halt(cpu);
            break;
        case KVM_EXIT_MMIO:
            // Guest 訪問未映射的記憶體，模擬設備
            handle_mmio(cpu);
            break;
    }
}
```

關鍵在於：**KVM 利用 CPU 的硬體虛擬化指令集（Intel VT-x / AMD-V），讓 Guest 指令可以直接在 CPU 上運行**，只有需要 Hypervisor 介入的指令才會退出。

### 3. Docker 的 Namespaces 隔離

Docker 底層使用 Linux 的 Namespaces API。以下是如何用程式碼建立隔離的行程：

```c
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    // 1. 建立新的 PID Namespace
    clone_flags = CLONE_NEWPID | CLONE_NEWNS | CLONE_NEWUTS | CLONE_NEWNET;
    
    // 2. 建立隔離的行程
    pid = clone(child_func, stack + STACK_SIZE, clone_flags, "container process");
    
    // 3. 在子行程中執行命令
    // 這行程會有自己的 PID 命名空間，看不到 Host 的行程
    // 也有自己的檔案系統 mount namespace
    // 也有自己的網路 namespace
}

int child_func(void *arg) {
    // 設定 hostname
    sethostname("mycontainer", 11);
    
    // mount 新的 rootfs
    mount("overlay", "/", NULL, MS_PRIVATE|MS_REC, NULL);
    chroot("/path/to/rootfs");
    chdir("/");
    
    // 執行應用程式
    execve("/bin/sh", args, envp);
}
```

### 4. Docker 的 Cgroups 資源限制

Cgroups（Control Groups）控制容器可以使用多少資源：

```bash
# 建立一個限制 CPU 和記憶體的 cgroup
# 限制 CPU 配額為 0.5 核心
echo 50000 > /sys/fs/cgroup/cpu/mycontainer/cpu.cfs_quota_us

# 限制記憶體為 512MB
echo 536870912 > /sys/fs/cgroup/memory/mycontainer/memory.limit_in_bytes

# 將行程加入 cgroup
echo $PID > /sys/fs/cgroup/cpu/mycontainer/tasks
```

用 Go 語言的視角：

```go
// libcontainer (Docker 底層) 建立 cgroup 的概念
func (c *cgroupManager) Apply(pid int) error {
    // 設定 CPU 限制
    if c.config.CpuQuota != 0 {
        path := filepath.Join(c.path, "cpu.cfs_quota_us")
        ioutil.WriteFile(path, []byte(strconv.FormatInt(c.config.CpuQuota, 10)), 0644)
    }
    
    // 設定記憶體限制
    if c.config.MemoryLimit != 0 {
        path := filepath.Join(c.path, "memory.limit_in_bytes")
        ioutil.WriteFile(path, []byte(strconv.FormatInt(c.config.MemoryLimit, 10)), 0644)
    }
    
    // 將行程加入 cgroup
    path := filepath.Join(c.path, "tasks")
    ioutil.WriteFile(path, []byte(strconv.Itoa(pid)), 0644)
    
    return nil
}
```

### 5. VirtualBox 的虛擬硬體模擬

VirtualBox 模擬一顆「虛擬磁碟控制器」的簡易概念：

```c
// VirtualBox 模擬 IDE 控制器的寫入
typedef struct {
    uint32_t base;
    uint8_t  selected_device;
    uint8_t  selected_register;
} IDEController;

uint8_t ide_read_register(IDEController *ide, uint8_t port) {
    switch (port) {
        case 0:  return ide->data_buffer[ide->data_index & 0x03];
        case 1:  return ide->error;      // Error register
        case 2:  return ide->sector_count;
        case 3:  return ide->sector_num;
        case 7:  return ide->status;     // Status register
    }
    return 0;
}

void ide_write_register(IDEController *ide, uint8_t port, uint8_t value) {
    switch (port) {
        case 0:  // Data register - 寫入 4 bytes
            memcpy(&ide->data_buffer, &value, 4);
            break;
        case 1:  // Feature register
            break;
        case 2:  // Sector count
            ide->sector_count = value;
            break;
        case 3:  // Sector number (LBA Low)
            ide->lba = (ide->lba & 0xFFFFFF00) | value;
            break;
        // ... 更多暫存器
    }
}
```

這就是為什麼 Guest OS 需要安裝「驅動程式」才能使用這些虛擬硬體——Guest 看到的磁碟控制器是 VirtualBox 模擬出來的。

## 核心差異對比表

| 特性 | VirtualBox | QEMU (+KVM) | Docker |
|------|------------|-------------|--------|
| 虛擬化層級 | 硬體模擬 | 硬體模擬/虛擬化 | 作業系統層級 |
| Guest OS | 必須有獨立 Kernel | 必須有獨立 Kernel | 共享 Host Kernel |
| 啟動時間 | 分鐘級 | 分鐘級 | 秒級 |
| 資源佔用 | 高 | 中~高 | 低 |
| 隔離程度 | 完全隔離 | 完全隔離 | 程序級隔離 |
| 主要用途 | 桌面測試 | 伺服器/嵌入式 | 應用部署 |
| 學習曲線 | 低 | 中~高 | 中 |

## 程式人何時該選什麼？

### 選 VirtualBox 當：
- 你需要在筆電上跑一個 Windows VM 測試 IE/Edge
- 你在學習 Linux，想要有一個安全的練習環境
- 你需要「圖形介面」來操作 VM

### 選 QEMU/KVM 當：
- 你在開發嵌入式系統，需要模擬 ARM 開發板
- 你在維護伺服器，需要高效的虛擬化
- 你需要更細粒度的硬體控制
- 你願意花時間學 libvirt/virsh

### 選 Docker 當：
- 你要部署服務，尤其是微服務架構
- 你需要快速建立一致的開發環境
- 你的 CI/CD pipeline 需要可重現的環境
- 你在意資源效率和啟動速度

## 小結：不是誰取代誰

這三個工具解決的是不同層次的問題。VirtualBox 解決「我想在桌上跑另一個 OS」的需求；QEMU 解決「我要高效且彈性的虛擬化」的需求；Docker 解決「如何確保我的程式在任何環境都能跑」的需求。

很多成熟的技術架構會把三者結合：用 VirtualBox 做開發測試、用 QEMU/KVM 跑生產環境的 VM、用 Docker 部署應用程式。它們是互補的，而不是互斥的。

理解這些工具的底層差異，能幫助你在面對不同問題時做出更聰明的技術選擇。這也是程式人該有的素養——不只會用工具，還要懂工具背後的原理。
